配置文件

TouchFish Developers2025/11/27大约 8 分钟文档配置

配置文件参考

本文档详细说明 TouchFish 服务器配置文件 config.json 的所有选项。

配置文件位置

配置文件 config.json 位于服务器程序的运行目录下。

TouchFish/
├── chat.exe          # 服务器程序
├── config.json       # 配置文件
├── log.txt           # 日志文件
└── ...

默认配置

服务器首次运行时会自动创建默认配置文件：

{
  "ban": {
    "words": [],
    "ip": [],
    "length": 2147483647
  },
  "ENTER_AFTER_PROMISE": false,
  "SHOW_ENTER_MESSAGE": false,
  "AUTO_REMOVE_OFFLINE": false
}

配置项详解

ban

封禁相关配置。

{
  "ban": {
    "words": ["敏感词1", "敏感词2"],
    "ip": ["192.168.1.110", "192.168.1.111"],
    "length": 500
  }
}

ban.words

类型：Array<string>
默认值：[]（空数组）
说明：敏感词列表

作用范围：

  1. ✅ 消息内容：包含敏感词的消息会被拒绝发送
  2. ✅ 用户名：包含敏感词的用户名会被拒绝加入
  3. ✅ 文件名：包含敏感词的文件名会被拒绝传输

示例：

{
  "ban": {
    "words": [
      "脏话",
      "广告",
      "不当内容",
      "spam"
    ]
  }
}

注意事项：

  • ⚠️ 敏感词区分大小写
  • ⚠️ 使用完全匹配，不支持正则表达式
  • ⚠️ 谨慎添加，避免误杀合法内容

通过命令修改：

# 添加敏感词
ban words add 敏感词

# 移除敏感词
ban words remove 敏感词

# 查看列表
ban words list

ban.ip

类型：Array<string>
默认值：[]（空数组）
说明：IP 黑名单

作用：

  • 黑名单中的 IP 地址无法连接服务器
  • 已连接的用户不会立即断开
  • 下次连接时会被拒绝

示例：

{
  "ban": {
    "ip": [
      "192.168.1.110",
      "192.168.1.111",
      "10.0.0.50"
    ]
  }
}

IP 格式：

  • ✅ IPv4 地址：192.168.1.100
  • ❌ 不支持 IP 段：192.168.1.*
  • ❌ 不支持 CIDR：192.168.1.0/24

通过命令修改：

# 封禁 IP
ban ip add 192.168.1.110

# 解封 IP
ban ip remove 192.168.1.110

# 查看列表
ban ip list

ban.length

类型：number
默认值：2147483647（最大整数）
说明：单条消息的最大字符数限制

单位：字符（不是字节）

推荐设置：

使用场景推荐值说明
普通聊天500-1000防止刷屏
技术交流1000-2000允许代码分享
无限制2147483647不限制长度

示例：

{
  "ban": {
    "length": 500
  }
}

效果：

  • 超过限制的消息会被拒绝
  • 客户端会收到 "Too long" 错误

通过命令修改：

# 设置为 500 字符
ban length 500

# 或
set ban_length 500

# 取消限制
ban length 2147483647

ENTER_AFTER_PROMISE

类型：boolean
默认值：false
说明：是否需要手动审批用户加入

false - 自动审批模式

{
  "ENTER_AFTER_PROMISE": false
}

行为：

  • ✅ 用户请求加入后立即接受
  • ✅ 无需管理员干预
  • ✅ 快速加入聊天室

适用场景：

  • 开放式聊天室
  • 学校机房
  • 公司内部网络
  • 信任环境

true - 手动审批模式

{
  "ENTER_AFTER_PROMISE": true
}

行为：

  • ⏳ 用户请求加入后需要等待审批
  • 👤 管理员使用 accept 或 reject 命令处理请求
  • 🔒 未经批准的用户无法进入

适用场景：

  • 私密聊天室
  • 需要控制成员的场景
  • 安全要求较高的环境

管理流程：

# 步骤 1：查看待审批请求
req

# 步骤 2：接受或拒绝
accept 0      # 接受第一个请求
reject 1      # 拒绝第二个请求
accept all    # 接受所有请求

通过命令修改：

# 开启手动审批
set ENTER_AFTER_PROMISE on

# 关闭手动审批
set ENTER_AFTER_PROMISE off

SHOW_ENTER_MESSAGE

类型：boolean
默认值：false
说明：是否广播用户加入消息

false - 静默加入

{
  "SHOW_ENTER_MESSAGE": false
}

行为：

  • 用户加入时不发送通知
  • 其他用户不知道有新成员加入
  • 保持聊天窗口清爽

适用场景：

  • 高频次加入/退出的环境
  • 不希望打扰现有用户
  • 大型聊天室

true - 广播加入

{
  "SHOW_ENTER_MESSAGE": true
}

行为：

  • 用户加入时向所有人广播
  • 显示类似 [系统] 小明 加入了聊天室
  • 增加互动感

适用场景：

  • 小型聊天室
  • 希望欢迎新成员
  • 社区氛围浓厚

客户端显示：

[系统提示] 小明 加入了聊天室

通过命令修改：

# 开启进入提示
set SHOW_ENTER_MESSAGE on

# 关闭进入提示
set SHOW_ENTER_MESSAGE off

AUTO_REMOVE_OFFLINE

类型：boolean
默认值：false
说明：是否自动移除长时间离线的用户

false - 保留用户记录

{
  "AUTO_REMOVE_OFFLINE": false
}

行为：

  • 用户断开连接后仍保留在用户列表
  • 用户记录（UID、用户名、发送次数等）保持
  • 再次连接时使用相同 UID

优点：

  • ✅ 保留用户历史数据
  • ✅ 便于统计和分析
  • ✅ 用户再次连接时身份连续

缺点：

  • ❌ 用户列表可能越来越长
  • ❌ 占用内存

true - 自动清理

{
  "AUTO_REMOVE_OFFLINE": true
}

行为：

  • 用户断开连接后自动从用户列表移除
  • 用户数据清除
  • 再次连接时分配新 UID

优点：

  • ✅ 保持用户列表简洁
  • ✅ 节省内存
  • ✅ 适合临时性聊天室

缺点：

  • ❌ 无法保留用户历史
  • ❌ 用户身份不连续

通过命令修改：

# 开启自动清理
set AUTO_REMOVE_OFFLINE on

# 关闭自动清理
set AUTO_REMOVE_OFFLINE off

配置文件管理

查看当前配置

方法 1：命令查看

192.168.1.100:8080> set show

输出完整的配置内容。

方法 2：直接打开文件

Windows：

notepad config.json

Linux/macOS：

cat config.json
# 或
nano config.json
# 或
vim config.json

修改配置

方法 1：通过命令（推荐）

使用 set 和 ban 命令动态修改：

# 修改配置项
set SHOW_ENTER_MESSAGE on
set ban_length 500

# 修改封禁列表
ban ip add 192.168.1.110
ban words add 脏话

优点：

  • ✅ 立即生效
  • ✅ 自动保存到文件
  • ✅ 不需要重启服务器

方法 2：直接编辑文件

  1. 停止服务器（exit 命令）
  2. 用文本编辑器打开 config.json
  3. 修改配置项
  4. 保存文件
  5. 重新启动服务器

优点：

  • ✅ 批量修改方便
  • ✅ 可以备份配置

缺点：

  • ❌ 需要重启服务器
  • ❌ JSON 格式错误会导致配置重置

备份配置

定期备份配置文件：

# Windows
copy config.json config.json.backup

# Linux/macOS
cp config.json config.json.backup

恢复配置

# Windows
copy config.json.backup config.json

# Linux/macOS
cp config.json.backup config.json

重置配置

删除配置文件后重启服务器，会自动创建默认配置：

# Windows
del config.json

# Linux/macOS
rm config.json

配置示例

示例 1：公开聊天室

场景：学校机房，开放使用，需要基本管理

{
  "ban": {
    "words": ["脏话", "广告"],
    "ip": [],
    "length": 500
  },
  "ENTER_AFTER_PROMISE": false,
  "SHOW_ENTER_MESSAGE": true,
  "AUTO_REMOVE_OFFLINE": true
}

特点：

  • 自动接受所有用户
  • 显示加入提示
  • 限制消息长度
  • 基本敏感词过滤
  • 自动清理离线用户

示例 2：私密聊天室

场景：私人聊天室，邀请制，成员固定

{
  "ban": {
    "words": [],
    "ip": [],
    "length": 2147483647
  },
  "ENTER_AFTER_PROMISE": true,
  "SHOW_ENTER_MESSAGE": true,
  "AUTO_REMOVE_OFFLINE": false
}

特点：

  • 手动审批用户
  • 显示加入提示
  • 不限制消息长度
  • 不过滤敏感词
  • 保留用户记录

示例 3：技术交流

场景：技术讨论，需要分享代码和公式

{
  "ban": {
    "words": [],
    "ip": [],
    "length": 2000
  },
  "ENTER_AFTER_PROMISE": false,
  "SHOW_ENTER_MESSAGE": true,
  "AUTO_REMOVE_OFFLINE": false
}

特点：

  • 自动接受用户
  • 较高的长度限制（支持代码）
  • 不过滤敏感词
  • 保留用户记录
  • 推荐使用 Astra 客户端（支持 Markdown 和 LaTeX）

示例 4：高安全环境

场景：需要严格管理的环境

{
  "ban": {
    "words": ["敏感词1", "敏感词2", "广告", "spam"],
    "ip": ["192.168.1.110", "192.168.1.111"],
    "length": 300
  },
  "ENTER_AFTER_PROMISE": true,
  "SHOW_ENTER_MESSAGE": true,
  "AUTO_REMOVE_OFFLINE": false
}

特点：

  • 手动审批所有用户
  • 严格的敏感词过滤
  • IP 黑名单
  • 较低的消息长度限制
  • 保留用户记录便于追踪

配置验证

服务器会在启动时验证配置文件：

有效配置

{
  "ban": {
    "words": ["test"],
    "ip": ["192.168.1.1"],
    "length": 500
  },
  "ENTER_AFTER_PROMISE": true,
  "SHOW_ENTER_MESSAGE": false,
  "AUTO_REMOVE_OFFLINE": true
}

✅ 通过验证：配置格式正确

无效配置

错误 1：JSON 格式错误

{
  "ban": {
    "words": ["test"],  # 注释不允许
    "ip": []
  }
}

❌ 验证失败：JSON 不支持注释

错误 2：类型错误

{
  "ban": {
    "words": "test",    // 应该是数组
    "ip": [],
    "length": "500"     // 应该是数字
  }
}

❌ 验证失败：数据类型不匹配

错误 3：缺少必需字段

{
  "ban": {
    "words": []
    // 缺少 ip 和 length
  }
}

❌ 验证失败：缺少必需字段

配置重置

如果配置文件无效，服务器会：

  1. 输出错误信息
  2. 自动重置为默认配置
  3. 覆盖原配置文件

配置最佳实践

安全建议

✅ 推荐做法：

  • 定期备份配置文件
  • 使用审批模式（私密环境）
  • 配置合理的敏感词列表
  • 封禁可疑 IP
  • 设置合理的消息长度限制

❌ 不推荐做法：

  • 从不备份配置
  • 不设置任何过滤规则
  • 过度封禁导致误伤
  • 不审查敏感词列表

性能建议

✅ 推荐做法：

  • 开启自动清理（AUTO_REMOVE_OFFLINE: true）
  • 限制消息长度
  • 定期审查用户列表

可维护性建议

✅ 推荐做法：

  • 使用命令修改配置（自动保存）
  • 添加注释（在文档中，不是配置文件中）
  • 记录配置变更历史

故障排查

配置不生效

可能原因：

  1. 配置文件格式错误
  2. 服务器未重启（手动编辑文件后）
  3. 缓存问题

解决方法：

  1. 验证 JSON 格式：使用在线 JSON 验证工具
  2. 重启服务器
  3. 使用命令修改配置而非手动编辑

配置被重置

可能原因：

  1. JSON 格式错误
  2. 数据类型不匹配
  3. 缺少必需字段

解决方法：

  1. 从备份恢复配置
  2. 检查配置文件格式
  3. 参考默认配置

无法保存配置

可能原因：

  1. 文件权限问题
  2. 磁盘空间不足
  3. 文件被锁定

解决方法：

  1. 检查文件权限
  2. 检查磁盘空间
  3. 关闭占用文件的程序
